The application follows a **client-server architecture** inside a **Turborepo monorepo**. The frontend and backend are fully decoupled, communicating over a REST API, with a shared package providing Zod schemas that enforce type safety at both boundaries.

There are now two distinct AI surfaces in the system:

- The in-app `/advisor` experience is served by `apps/api`, which builds a live finance context from MongoDB and calls Gemini for grounded chat plus confirmation-based action planning.
- The separate `agentic-ai/` service remains Claude-powered and MCP-backed for specialist agent workflows.

| Task    | `dependsOn`                    | Outputs                  | Cached |
| ------- | ------------------------------ | ------------------------ | ------ |
| `build` | `^build` (shared-types first)  | `.next/**`, `dist/**`    | Yes    |
| `dev`   | -                              | -                        | No     |
| `lint`  | -                              | -                        | Yes    |
| `test`  | -                              | -                        | Yes    |
| `clean` | -                              | -                        | No     |

`apps/web` and `apps/api` both depend on `packages/shared-types`. Turbo builds shared-types first, then builds both apps in parallel. `mcp/` and `agentic-ai/` are independent packages that do not depend on shared-types; `agentic-ai/` connects to the MCP server at runtime.

## API Architecture

### Layered Design

```mermaid
graph LR
    REQ([HTTP Request]) --> R[Route]
    R --> MW1[Auth Middleware]
    MW1 --> MW2[Validate Middleware]
    MW2 --> C[Controller]
    C --> S[Service]
    S --> M[Mongoose Model]
    M --> DB[(MongoDB)]
    DB --> M
    M --> S
    S --> C
    C --> RES([HTTP Response])

    ERR[Error Handler] -.-> RES
    MW1 -. "401" .-> ERR
    MW2 -. "400" .-> ERR
    S -. "4xx/5xx" .-> ERR

    style REQ fill:#6366f1,stroke:#000,color:#fff
    style RES fill:#10b981,stroke:#000,color:#fff
    style ERR fill:#ef4444,stroke:#000,color:#fff
    style DB fill:#47a248,stroke:#000,color:#fff
```

| Layer           | Responsibility                                  | Location           |
| --------------- | ----------------------------------------------- | ------------------ |
| **Routes**      | URL mapping, Swagger JSDoc annotations           | `src/routes/`      |
| **Middleware**   | Auth, validation, CORS, rate limiting, errors   | `src/middleware/`   |
| **Controllers** | Parse request, call service, format response     | `src/controllers/` |
| **Services**    | Business logic, data access, aggregations        | `src/services/`    |
| **Models**      | Mongoose schemas, indexes, virtuals              | `src/models/`      |
| **Utils**       | ApiError class, async handler, pagination        | `src/utils/`       |

### In-App AI Financial Advisor Flow

```mermaid
sequenceDiagram
    participant U as User
    participant WEB as Next.js /advisor
    participant API as Express API
    participant AUTH as JWT Auth
    participant ADV as advisor.service
    participant DB as MongoDB
    participant GEM as Gemini API

    U->>WEB: Ask for analysis or a create action
    WEB->>API: POST /api/v1/advisor/chat
    API->>AUTH: Verify bearer token
    AUTH-->>API: userId
    API->>ADV: chat(userId, message, history)
    ADV->>DB: Load accounts, transactions, categories, budgets, goals, recurring rules
    DB-->>ADV: Live finance data
    ADV->>GEM: Structured prompt + finance context + frontend workflow guide
    GEM-->>ADV: reply-only JSON
    ADV-->>WEB: grounded reply with analysis or exact in-app steps
```

The advisor never executes model-proposed actions automatically. Gemini can only propose structured create actions, and the frontend must explicitly confirm them before the API calls the underlying account, category, budget, goal, recurring, or transaction service.

### Error Handling

All errors flow through a single global error handler. The custom `ApiError` class provides factory methods:

```mermaid
graph LR
    subgraph Sources["Error Sources"]

    ZE --> NORM
    ME --> NORM
    AE --> NORM
    UE --> NORM

    NORM --> R400["400 Bad Request"]
    NORM --> R401["401 Unauthorized"]
    NORM --> R404["404 Not Found"]
    NORM --> R409["409 Conflict"]
    NORM --> R500["500 Internal"]

    style R400 fill:#f59e0b,stroke:#000,color:#000
```

**Response shape:** `{ success: false, error: { code, message, details? } }`

## Database Schema

### Entity Relationship Diagram

```mermaid
erDiagram
    USER ||--o{ ACCOUNT : owns
    USER ||--o{ TRANSACTION : owns
    USER ||--o{ CATEGORY : creates
    USER ||--o{ BUDGET : sets
    USER ||--o{ GOAL : tracks
    USER ||--o{ RECURRING_RULE : defines

    ACCOUNT ||--o{ TRANSACTION : contains
    CATEGORY ||--o{ TRANSACTION : categorizes
    CATEGORY ||--o{ BUDGET : "spending limit for"
    ACCOUNT ||--o{ RECURRING_RULE : "charges to"
    CATEGORY ||--o{ RECURRING_RULE : categorizes

    USER {
        ObjectId _id PK
        string email UK
        string name
        string passwordHash
        string avatarUrl
        string currency
    }

    ACCOUNT {
        ObjectId _id PK
        ObjectId userId FK
        string name
        enum type "checking|savings|credit_card|cash|investment"
        number balance
        string color
        boolean isArchived
    }

    TRANSACTION {
        ObjectId _id PK
        ObjectId userId FK
        ObjectId accountId FK
        ObjectId categoryId FK
        enum type "income|expense|transfer"
        number amount
        string description
        date date
        string[] tags
    }

    CATEGORY {
        ObjectId _id PK
        ObjectId userId FK "null = system default"
        string name
        string icon
        string color
        enum type "income|expense"
        boolean isDefault
    }

    BUDGET {
        ObjectId _id PK
        ObjectId userId FK
        ObjectId categoryId FK
        number amount
        enum period "monthly|weekly"
        number alertThreshold
        boolean isActive
    }

    GOAL {
        ObjectId _id PK
        ObjectId userId FK
        string name
        number targetAmount
        number currentAmount
        date deadline
        boolean isCompleted
    }

    RECURRING_RULE {
        ObjectId _id PK
        ObjectId userId FK
        ObjectId accountId FK
        ObjectId categoryId FK
        enum frequency "daily|weekly|biweekly|monthly|yearly"
        number amount
        date nextDueDate
        boolean isActive
    }
```

All user-owned entities store a `userId` foreign key. Ownership is enforced at the service layer - every query filters by `userId` from the JWT.

### State Management

| Concern      | Solution                                                    |
| ------------ | ----------------------------------------------------------- |

| Theme        | next-themes (class-based, `localStorage`)                   |

| URL state    | Next.js `searchParams` for filters                          |

No global client-side store (Redux, Zustand, etc.) is used. TanStack Query serves as the single source of truth for all server-derived state, while lightweight client preferences such as appearance and motion settings are persisted locally.
