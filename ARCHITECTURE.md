The application follows a **client-server architecture** inside a **Turborepo monorepo**. The frontend and backend are fully decoupled, communicating over a REST API, with a shared package providing Zod schemas that enforce type safety at both boundaries.

```mermaid
graph LR
    ST["packages/shared-types"] --> API["apps/api"]

```

| Task    | `dependsOn`                    | Outputs                  | Cached |
| ------- | ------------------------------ | ------------------------ | ------ |
| `build` | `^build` (shared-types first)  | `.next/**`, `dist/**`    | Yes    |
| `dev`   | -                              | -                        | No     |
| `lint`  | -                              | -                        | Yes    |
| `test`  | -                              | -                        | Yes    |
| `clean` | -                              | -                        | No     |

`apps/web` and `apps/api` both depend on `packages/shared-types`. Turbo builds shared-types first, then builds both apps in parallel.

## API Architecture

### Layered Design

```mermaid
graph LR
    REQ([HTTP Request]) --> R[Route]
    R --> MW1[Auth Middleware]
    MW1 --> MW2[Validate Middleware]
    MW2 --> C[Controller]
    C --> S[Service]
    S --> M[Mongoose Model]
    M --> DB[(MongoDB)]
    DB --> M
    M --> S
    S --> C
    C --> RES([HTTP Response])

    ERR[Error Handler] -.-> RES
    MW1 -. "401" .-> ERR
    MW2 -. "400" .-> ERR
    S -. "4xx/5xx" .-> ERR

    style REQ fill:#6366f1,stroke:#000,color:#fff
    style RES fill:#10b981,stroke:#000,color:#fff
    style ERR fill:#ef4444,stroke:#000,color:#fff
    style DB fill:#47a248,stroke:#000,color:#fff
```

| Layer           | Responsibility                                  | Location           |
| --------------- | ----------------------------------------------- | ------------------ |
| **Routes**      | URL mapping, Swagger JSDoc annotations           | `src/routes/`      |
| **Middleware**   | Auth, validation, CORS, rate limiting, errors   | `src/middleware/`   |
| **Controllers** | Parse request, call service, format response     | `src/controllers/` |
| **Services**    | Business logic, data access, aggregations        | `src/services/`    |
| **Models**      | Mongoose schemas, indexes, virtuals              | `src/models/`      |
| **Utils**       | ApiError class, async handler, pagination        | `src/utils/`       |

### Middleware Pipeline

```mermaid
graph TD
    REQ([Incoming Request]) --> CORS["1. CORS<br/><i>configurable origin</i>"]
    CORS --> MORGAN["2. Morgan<br/><i>HTTP logging</i>"]
    MORGAN --> JSON["3. Express JSON<br/><i>body parsing, 10MB limit</i>"]
    JSON --> RATE["4. Rate Limiter<br/><i>auth endpoints only</i>"]
    RATE --> AUTH["5. Auth Middleware<br/><i>verify JWT Bearer token</i>"]
    AUTH --> VAL["6. Validate Middleware<br/><i>Zod schema on body/query/params</i>"]
    VAL --> HANDLER["7. Route Handler"]
    HANDLER --> ERRH["Error Handler<br/><i>global catch-all</i>"]
    ERRH --> RES([Response])

    AUTH -. "unauthorized" .-> ERRH
    VAL -. "validation error" .-> ERRH

    style REQ fill:#6366f1,stroke:#000,color:#fff
    style RES fill:#10b981,stroke:#000,color:#fff
    style ERRH fill:#ef4444,stroke:#000,color:#fff
```

### Error Handling

All errors flow through a single global error handler. The custom `ApiError` class provides factory methods:

```mermaid
graph LR

        UE["Unhandled Error"]
    end

    subgraph Handler["Global Error Handler"]
        NORM["Normalize to standard shape"]
    end

    ZE --> NORM
    ME --> NORM
    AE --> NORM
    UE --> NORM

    NORM --> R400["400 Bad Request"]
    NORM --> R401["401 Unauthorized"]
    NORM --> R404["404 Not Found"]
    NORM --> R409["409 Conflict"]
    NORM --> R500["500 Internal"]

    style R400 fill:#f59e0b,stroke:#000,color:#000
    style R401 fill:#ef4444,stroke:#000,color:#fff
```

**Response shape:** `{ success: false, error: { code, message, details? } }`

### Pagination

Every list endpoint supports:

| Param   | Default | Max |
| ------- | ------- | --- |
| `page`  | 1       | -   |
| `limit` | 20      | 100 |

Response includes `{ data, pagination: { page, limit, total, totalPages } }`.

## Database Schema

### Entity Relationship Diagram

```mermaid
erDiagram
    USER ||--o{ ACCOUNT : owns
    USER ||--o{ TRANSACTION : owns
    USER ||--o{ CATEGORY : creates
    USER ||--o{ BUDGET : sets
    USER ||--o{ GOAL : tracks
    USER ||--o{ RECURRING_RULE : defines

    ACCOUNT ||--o{ TRANSACTION : contains
    CATEGORY ||--o{ TRANSACTION : categorizes
    CATEGORY ||--o{ BUDGET : "spending limit for"
    ACCOUNT ||--o{ RECURRING_RULE : "charges to"
    CATEGORY ||--o{ RECURRING_RULE : categorizes

    USER {
        ObjectId _id PK
        string email UK
        string name
        string passwordHash
        string avatarUrl
        string currency
    }

    ACCOUNT {
        ObjectId _id PK
        ObjectId userId FK
        string name
        enum type "checking|savings|credit_card|cash|investment"
        number balance
        string color
        boolean isArchived
    }

    TRANSACTION {
        ObjectId _id PK
        ObjectId userId FK
        ObjectId accountId FK
        ObjectId categoryId FK
        enum type "income|expense|transfer"
        number amount
        string description
        date date
        string[] tags
    }

    CATEGORY {
        ObjectId _id PK
        ObjectId userId FK "null = system default"
        string name
        string icon
        string color
        enum type "income|expense"
        boolean isDefault
    }

    BUDGET {
        ObjectId _id PK
        ObjectId userId FK
        ObjectId categoryId FK
        number amount
        enum period "monthly|weekly"
        number alertThreshold
        boolean isActive
    }

    GOAL {
        ObjectId _id PK
        ObjectId userId FK
        string name
        number targetAmount
        number currentAmount
        date deadline
        boolean isCompleted
    }

    RECURRING_RULE {
        ObjectId _id PK
        ObjectId userId FK
        ObjectId accountId FK
        ObjectId categoryId FK
        enum frequency "daily|weekly|biweekly|monthly|yearly"
        number amount
        date nextDueDate
        boolean isActive
    }
```

All user-owned entities store a `userId` foreign key. Ownership is enforced at the service layer - every query filters by `userId` from the JWT.

### Client Data Flow

```mermaid
sequenceDiagram
    participant C as React Component
    participant H as TanStack Query Hook
    participant AC as API Client

    API->>DB: Transaction.find({ userId })
    DB-->>API: documents[]
    API-->>AC: { success: true, data, pagination }
    AC-->>H: response data
    H-->>C: { data, isLoading, error }

    Note over C,H: Mutations trigger cache<br/>invalidation + refetch

    C->>H: useCreateTransaction()
    H->>AC: apiClient.post("/transactions", body)
    AC->>API: POST /api/v1/transactions
    API->>DB: Transaction.create(...)
    DB-->>API: new document
    API-->>AC: { success: true, data }
    AC-->>H: response
    H->>H: invalidateQueries(["transactions"])
    H-->>C: Toast: "Transaction created"
```

### State Management

| Concern      | Solution                                                    |
| ------------ | ----------------------------------------------------------- |
| Server state | TanStack Query 5 (cache, refetch, optimistic updates)       |
| Form state   | React Hook Form + Zod resolver                              |
| Auth session | NextAuth.js (JWT strategy)                                  |
| Theme        | next-themes (class-based, `localStorage`)                   |
| URL state    | Next.js `searchParams` for filters                          |

No global client-side store (Redux, Zustand, etc.) is used. TanStack Query serves as the single source of truth for all server-derived state.

## Authentication Flow

### Full Auth Sequence

```mermaid
sequenceDiagram
    participant U as User
    participant WEB as Next.js Frontend
    participant NA as NextAuth.js
    participant API as Express API
    participant DB as MongoDB

    rect rgb(30, 41, 59)
        Note over U,DB: Registration
        U->>WEB: Fill register form
        WEB->>API: POST /api/v1/auth/register<br/>{email, name, password}
        API->>API: Validate with Zod
        API->>API: Hash password (bcrypt 12 rounds)
        API->>DB: Create User document
        DB-->>API: User created
```

**Schemas** define runtime validation rules. **Types** are inferred from schemas using `z.infer<>`, ensuring they can never drift.

Enums, constraints (min/max length, email format, positive numbers), and optional fields are all defined once and consumed by both the API validation middleware and the frontend form resolvers.
